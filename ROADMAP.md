# Rust-IME 架构演进路线图 (Architecture Roadmap)

本项目目前已完成从 0 到 1 的原型开发，实现了基于 TSF 的进程外输入法核心逻辑。为了将项目推向更高水平的工程化和稳定性，未来的工作重点将从“功能堆砌”转向“架构治理”。

---

## 📅 阶段一：基础加固与观测 (Foundation & Stability)
*目标：提升系统透明度，规范底层通信，减少低级同步 Bug。*

- [ ] **结构化日志系统**：
    - 引入 `tracing` 或 `log4rs` 框架替代原始的 `println!`。
    - 支持日志分级（Info/Debug/Error）并持久化到本地文件 `%APPDATA%/rust_ime/logs/`。
    - **收益**：在无控制台环境下也能追踪崩溃和异常。
- [ ] **IPC 协议规范化**：
    - 引入 `serde` 和 `bincode` (二进制) 或 `json`。
    - 将 `tsf.rs` 中原始的字节数组通信改为强类型的 `IpcMessage` 枚举。
    - **收益**：消除手动计算偏移量带来的潜在内存越界和逻辑错误。
- [ ] **健壮的错误处理**：
    - 全面审计并减少 `unwrap()` / `expect()` 的使用，建立统一的 `AppError`。
    - 增加主循环的 Panic 恢复机制，防止 UI 异常拖垮输入服务。

---

## 📅 阶段二：状态管理与并发重构 (State & Concurrency)
*目标：解决多线程状态同步冲突，提升 UI 响应速度。*

- [ ] **单一数据源 (SSoT) 架构**：
    - 建立全局唯一的 `AppState` 状态机。
    - UI 线程（Slint/Tray）改为“观察者模式”，仅负责渲染状态，不直接修改业务逻辑。
    - **收益**：彻底杜绝“显隐逻辑冲突”、“中英文状态不一”等同步 Bug。
- [ ] **解耦主循环 (God Object Removal)**：
    - 将 `main.rs` 的职责拆分为：`IpcService`, `GuiService`, `ConfigService`, `TrayService`。
    - 实现各模块的独立生命周期管理（Start/Stop/Reload）。
- [ ] **消息驱动 (Actor-like) 模型**：
    - 将 `Processor` 放入独立线程，通过 `mpsc` 频道接收按键。
    - 移除频繁的 `Mutex<Processor>` 锁定。
    - **收益**：UI 渲染即使卡顿（如加载大图）也绝不影响用户的打字输入流畅度。

---

## 📅 阶段三：核心引擎与存储优化 (Engine & Storage)
*目标：降低内存占用，提升超大规模词库下的检索效率。*

- [ ] **分层词库存储方案**：
    - 核心高频词（前 10W）保留在内存 Trie 树中。
    - 扩展词库和历史记录切换为 SQLite 或类似的高效本地键值对数据库。
    - **收益**：内存占用可降低 50% 以上，支持百万级词库。
- [ ] **输入处理流水线 (Pipeline)**：
    - 将输入处理流程标准化：`切分(Segment) -> 检索(Search) -> 过滤(Filter) -> 排序(Sort) -> 联想(Suggest)`。
    - **收益**：易于增加新功能（如：智能纠错、模糊音、云联想）。

---

## 📅 阶段四：高级特性与生态 (Ecosystem & Extension)
*目标：提升用户体验的精致感和扩展性。*

- [ ] **Web 配置中心热加载**：
    - 实现配置变更后的实时应用，无需重启程序。
    - 增加可视化的外观/皮肤在线预览与导出。
- [ ] **多端同步与云支持 (可选)**：
    - 提供插件接口，支持 WebDAV 或其他云端同步用户词库。
- [ ] **多系统适配增强**：
    - 在 Linux 下基于 GTK Layer Shell 实现更完美的桌面交互。

## 📅 阶段五：高度模块化与多语言引擎 (Modularization & Multi-language)
*目标：打破“硬编码”枷锁，实现输入法方案的“即插即用”，轻松支持韩语等新语种。*

- [ ] **输入方案接口化 (InputScheme Trait)**：
    - 定义统一的 `InputScheme` Trait，将拼音、笔画、日语、英语等逻辑彻底解耦。
    - **收益**：新增语种（如韩语）时，只需实现该 Trait，无需修改核心检索逻辑。
- [ ] **三段式输入流水线 (Pipeline)**：
    - **预处理 (Pre-processor)**：处理双拼转全拼、1-5数字转字母、韩语组字逻辑。
    - **检索层 (Matcher)**：通用的高性能 Trie 树检索。
    - **后处理 (Post-processor)**：处理英文辅助码、笔画提示、自动上屏等过滤逻辑。
- [ ] **数据驱动逻辑 (Schema-Driven)**：
    - 引入 `schema.json` 规范。通过配置文件定义按键映射、转换规则和词典路径，而非在 Rust 中硬编码 `if is_xxx`。
- [ ] **组字状态机 (Composition State Machine)**：
    - 针对韩语 (Hangul) 等需要将字母组合成方块字的语言，建立专门的组字状态维护机制，使 `buffer` 具备更灵活的展示能力。

---

> **架构师寄语**：好的代码是演化出来的，不是一次性设计出来的。先保持项目能跑，再通过局部的重构让它跑得更好。
